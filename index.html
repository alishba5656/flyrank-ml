Abstract

5-Sentence Summary

Question: Can search performance signals from an early time window predict which pages will grow, decline, recover, or need review in a future window — and produce an actionable ranked content engine? Method: Using the FlyRank ML Internship dataset, I engineered 12 time-aware features from page-level GSC aggregates, applied a strict early-window feature / late-window label split to prevent leakage, and trained a Gradient Boosting classifier against a majority-class baseline. Result: The model achieved an F1-macro of 0.71 on 5-fold cross-validation, compared to a baseline of 0.18 — a 3.9× improvement — with click_momentum, clicks_per_week, and avg_position emerging as the top predictors. Output: Every page in the dataset receives a ranked opportunity score and a reason code mapping to one of four action types: PROTECT, REFRESH, BOOST, or REVIEW. Framing: Results are directional and observational; this model supports content prioritisation decisions and should be used alongside editorial judgment, not in place of it.
Introduction

The Content Prioritisation Problem

Content teams managing large web properties face a recurring prioritisation problem: with hundreds or thousands of pages, which ones deserve attention this sprint? The default answer — sort by traffic and work top-down — systematically ignores two high-value categories: declining pages that could be rescued, and recovering pages where a small investment would accelerate momentum that is already building.

This paper addresses the decision: given a snapshot of page-level search performance, which pages should be acted on first, and what action should be taken? The answer is a scored, ranked action engine that classifies each page into one of four content states — GROWING, DECLINING, RECOVERING, or REVIEW — and attaches a reason code that maps directly to editorial action.

The model is trained on the FlyRank ML Internship dataset, a real-world search performance dataset containing page-level Google Search Console signals aggregated by week.

Methodology

Assumptions, Features, Labels & Validation

Time-Aware Split Design

The dataset is split at the 60th percentile date. All features are derived from the early window only (weeks 1–60%). Labels are assigned from late window only (weeks 61–100%). This design prevents leakage: the model never sees future data during training.

# Time-aware split — no leakage cutoff = df['date_week'].quantile(0.60) early = df[df['date_week'] <= cutoff] # features from here late = df[df['date_week'] > cutoff] # labels from here

Label Definition

LabelRuleMeaning
GROWING click_growth > +20% AND position improved or stable (Δpos ≤ 2) Strong performer with positive momentum
DECLINING click_growth < −20% Losing traffic — intervention needed
RECOVERING click_growth > +10% AND early clicks below median Building from a low base — reinforce momentum
REVIEW All other pages Flat, volatile, or unclear signal — needs editorial review

Validation Design

Primary: 5-fold stratified cross-validation on F1-macro. Stratified to preserve label balance across folds.

Secondary: Time-ordered holdout — final 20% of pages (sorted by earliest active week) held out for confusion matrix evaluation.

Leakage checks: (1) No late-window columns in feature matrix. (2) click_growth and pos_change used only for label construction, never as model features. (3) Split confirmed via date range inspection before model training.

Baseline: Majority-class DummyClassifier — predicts the most frequent label for every page.

Limitations & Honest Framing

What This Model Can and Cannot Say

Observational, not causal. The model identifies pages that are associated with future decline or growth patterns — it does not establish that any feature causes those outcomes. Position changes may reflect Google algorithm shifts, competitor activity, or seasonality that the model cannot distinguish.

Label quality is rule-based. GROWING / DECLINING / RECOVERING / REVIEW labels are derived from thresholds I defined (+20%, −20%, +10%). Different thresholds produce different label distributions and different model behaviour. The 20% threshold was chosen based on dataset distribution inspection, not external ground truth.

REVIEW class is heterogeneous. Pages that are flat, volatile, and inconsistent all land in REVIEW. The model's lower precision on this class reflects this: it is a catch-all, not a clean concept. Treat REVIEW scores as "further investigation warranted," not "this page has a specific problem."

Time window sensitivity. Results reflect the date range in this dataset. Pages with seasonal content may be misclassified depending on where the train/label split falls relative to their seasonal peak.

No content signals. The model uses only GSC performance metrics. It has no access to content quality, page freshness, word count, structured data, or backlink profile. A page declining due to content staleness looks identical to one declining due to competition.

Use as decision support, not authority. Output scores should be read as directional priority signals. Editorial teams should review flagged pages with context the model cannot access before committing to action.

Ranked Recommendations

The Action Playbook

Pages are scored 0–1 (opportunity_score). Higher = more urgent or higher potential. Use the table below to map predicted label to editorial action. The model outputs a reason code per page — these map directly to the four action types.

Priority Label Score Range Recommended Action Why Now
1 — Urgent DECLINING 0.70 – 1.00 REFRESH — audit content freshness, check backlink profile, review search intent alignment Click loss already underway — delay increases the recovery cost
2 — High RECOVERING 0.45 – 0.70 BOOST — add internal links, expand content depth, check for featured snippet opportunity Momentum is building from a low base — reinforce it before it plateaus
3 — Maintain GROWING 0.20 – 0.45 PROTECT — monitor weekly, expand with related content cluster, do not change what's working Strong signal — risk is in over-editing and breaking what works
4 — Backlog REVIEW 0.00 – 0.45 MONITOR — assess search intent, check for cannibalisation, consider merge or prune Ambiguous signal — don't invest until editorial review clarifies the problem
How to use the ranked output
Sort the ranked_opportunities.csv output by opportunity_score descending. Work through DECLINING pages first this sprint, RECOVERING second. Do not act on REVIEW pages until a human has reviewed the page's search intent alignment. Never act on GROWING pages without a clear reason.
